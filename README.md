## Middleware

В проекте реализованы следующие middleware:

1. **Логирование запросов** - выводит в консоль HTTP-метод, путь и статус ответа
2. **Добавление кастомного заголовка** - добавляет заголовок `X-Powered-By: ASP.NET Core Lab27`
3. **Проверка ключа доступа** - проверяет наличие параметра `?key=secret`, возвращает 401 при отсутствии

## Ответы на вопросы

### 1. Чем Minimal API отличается от MVC в ASP.NET Core?

**Minimal API:**
- Минималистичный подход без контроллеров
- Весь код в Program.cs
- Идеально для микросервисов и простых API
- Меньше кода и настроек
- Быстрее разработка небольших проектов
- Нет атрибутов маршрутизации

**MVC (Model-View-Controller):**
- Структурированный подход с контроллерами
- Разделение логики по классам-контроллерам
- Подходит для больших приложений
- Поддержка представлений (Views)
- Больше возможностей для валидации, фильтрации
- Лучшая организация кода в больших проектах

### 2. Что произойдёт, если поставить app.MapGet(...) до app.Use(...)?

Если зарегистрировать маршруты (MapGet) **до** middleware, то middleware **не будет применяться** к этим маршрутам. Middleware работает только с запросами, которые проходят через него. 

Пример:

```csharp
    // НЕПРАВИЛЬНО:
    app.MapGet("/", () => "Hello");  // Этот маршрут НЕ пройдёт через middleware
    app.Use(async (context, next) => { ... });  // Middleware не сработает для "/"

    // ПРАВИЛЬНО:
    app.Use(async (context, next) => { ... });  // Middleware регистрируется первым
    app.MapGet("/", () => "Hello");  // Теперь запрос пройдёт через middleware
```

### 3. Почему ASP.NET Core преобразует PascalCase в camelCase при сериализации JSON?
Это делается для совместимости с JavaScript-экосистемой:

- JavaScript конвенция: в JavaScript принято использовать camelCase для имён свойств (это стандарт кодирования)
- Совместимость: большинство JavaScript-библиотек и фреймворков ожидают camelCase
- JSON стандарт: хотя JSON не требует camelCase, это стало де-факто стандартом в веб-разработке
- Frontend интеграция: упрощает работу с данными на клиенте (React, Angular, Vue)

### 4. Что означает код ответа HTTP 401? А 404? А 200?
200 OK - успешный запрос. Сервер обработал запрос и вернул данные.

401 Unauthorized - не авторизован. Клиент не предоставил правильные учётные данные для доступа к ресурсу. Требуется аутентификация.

404 Not Found - ресурс не найден. Сервер работает, но запрашиваемый URL не существует.

Другие распространённые коды:

- 403 Forbidden - доступ запрещён (аутентификация прошла, но нет прав)
- 500 Internal Server Error - ошибка на сервере
- 400 Bad Request - неверный запрос

### 5. Чем dotnet run отличается от dotnet watch run?

dotnet run:

- Запускает приложение один раз
- При изменениях в коде нужно вручную остановить (Ctrl+C) и перезапустить
- Меньше потребляет ресурсов
- Подходит для production-сборки

dotnet watch run:
- Запускает приложение с горячей перезагрузкой (hot reload)
- Автоматически перезапускает сервер при изменении файлов
- Ускоряет разработку (не нужно постоянно перезапускать вручную)
- Идеально для разработки
- Немного больше потребляет ресурсов
